## 4. 화면정의 상세 (SCR-01~09)

> 각 화면은 ① 개요 ② 화면 구성 ③ 상태·엣지케이스 ④ 데이터·권한의 4단 구조. (화면정의5 전문을 v3.0 확정본으로 승계)

### SCR-01. /program — 과정 안내 [U]

**① 개요** — "따라갈 수 있을까"(US-01)와 "다른 과정과 뭐가 다른가"(US-02)를 지원 전에 해소하고 지원 CTA로 보낸다. 비로그인 방문자 대상.

**② 화면 구성** — 1) 히어로: 포지셔닝 문장(경쟁사_비교분석2의 채택안) + 모집 상태 배지 + 지원 CTA 2) 커리큘럼 개요(수료 기준 = 자기 회사 문제를 푼 결과물 명시) 3) **자가진단(신규)**: 5문항(코딩 경험·가용 시간·AX 상황 A/B/C 판별), 결과 카드 + 맞춤 문구 + 지원 CTA 4) **수료생 Build 미리보기(신규)**: 공개 동의분 3건 카드 5) **경쟁 비교 섹션(신규)**: 최고위과정·플랫폼 강의·유튜브 대비 3축(실습·CEO 의사결정·수료 후 지속) 6) FAQ + 하단 CTA

**③ 상태·엣지케이스** — 모집 마감 시 CTA를 "다음 기수 알림 신청"으로 전환 · 자가진단 응답은 비저장(비로그인 개인정보 미수집) · Build 공개 동의분 3건 미만이면 섹션 숨김

**④ 데이터·권한** — 자가진단은 정적 콘텐츠, 결과 유형만 `application.submitted` 이벤트 속성으로 전달 · Build 미리보기는 `builds.public_consent = true`만 노출

### SCR-02. /apply/status — 전형 상태 조회 [U · P0 승격]

**① 개요** — 로그인 없이 매직링크로 전형 진행 상황을 확인시켜 이탈·불안을 줄인다. 접수 확인 알림톡·메일의 매직링크로만 진입.

**② 화면 구성** — 1) 지원자 이름·기수 헤더 2) 3단계 스테퍼(접수완료→심사중→결과발표) 3) 단계별 안내문 + 다음 단계 예상 일정 4) 합격 시 등록 안내 요약 + 포털 온보딩 링크

**③ 상태·엣지케이스** — 매직링크 만료(기본 7일) 시 재발송 · 위조 토큰은 정보 노출 없이 안내 페이지로 · 알림톡(T-01~03)과 화면 표시 상태는 항상 일치(전환 트리거는 /admin/applications)

**④ 데이터·권한** — `applications.status` 읽기 전용 · 서명된 일회성 매직링크(지원 건 단위) · 개인정보는 이름 외 미노출

### SCR-03. /portal/billing — 인보이스·결제 위임 [U]

**① 개요** — CEO가 결제 실무를 연결 assistant에게 넘기고 진행 상태만 확인. 합격 CEO 본인 + 위임받은 assistant.

**② 화면 구성** — 1) 인보이스 카드(금액·계좌·기한·입금 상태) 2) 세금계산서 요청 폼 3) **위임 패널(신규)**: assistant 목록 + 위임 토글 + 위임 상태 배지 4) 처리 이력 타임라인(actor 기록)

**③ 상태·엣지케이스** — assistant 미연결 시 연결 안내로 유도 · 위임 해제 시 즉시 접근 차단 + 진행 건 CEO 승계 알림 · 입금 확인은 manual 운영 승계(화면은 상태 표시만)

**④ 데이터·권한** — `invoices` + 세금계산서 요청. RLS: 본인 + 위임 flag 켜진 연결 assistant + admin · 위임 변경은 CEO만

### SCR-04. /portal/ai — AI 조교 + 막힘 에스컬레이션 [U]
