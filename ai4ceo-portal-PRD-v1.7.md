## v1.7 변경 요약 (Changelog)

v1.7은 강의를 수강하고 수료한 사람들이 원할 경우 자신과 회사를 소개할 수 있는 **선택형 공개 프로필** 기능을 추가한다. 이 기능은 완전히 optional이며, 기본값은 비공개다. 수료생은 아무것도 올리지 않아도 되고, 원하면 사진, 이력, 회사 설명, 연락/협업 관심사 등을 입력해 동문 또는 전체 공개 범위로 노출할 수 있다.

# 1. 제품 개요

## 1.1 배경

AI4CEO는 일반 기업의 CEO와 임원을 대상으로 하는 바이브코딩(Claude Code 기반 코딩·디자인) 스쿨이다. 현재 모집, 안내, 교육 운영, 수료 후 관리가 이메일·전화·오프라인 자료로 분산되어 있어, 지원자 경험과 운영 효율 모두에 한계가 있다. 본 포탈은 모집(Acquisition) → 과정안내(Information) → 수강신청(Application) → 교육진행(Learning/CMS) → **결과물 산출(Build)** → 수료 후 AS(Retention) → AI 의사결정 브리프(Content)의 전체 수명주기를 하나의 웹 포탈로 통합한다.

## 1.2 제품 비전

CEO가 지원서를 내는 순간부터, **자기 손으로 첫 결과물을 만들어 회사에 적용하고**, 수료 후 동문이 되어 최신 AI 의사결정 브리프를 받아보는 순간까지, 모든 접점이 카카오톡 알림과 함께 끊김 없이 이어지는 경영진 전용 AI 교육 플랫폼.

## 1.3 핵심 설계 원칙

1. **결과물 우선(Outcome over Completion)**: 성공의 기준은 출석·수료가 아니라 "수강생이 실제로 만든 결과물과 회사 적용"이다. 모든 모듈과 KPI는 이 기준에 정렬한다.
2. **모바일 퍼스트, 앱 설치 불필요**: 대상 사용자(CEO·임원)는 앱 설치 요구 자체가 이탈 요인이다. 반응형 웹 + PWA로 제공하고, 알림은 카카오 알림톡으로 해결한다.
3. **로그인 장벽 최소화**: 지원 단계는 비로그인으로 완료 가능해야 한다. 계정은 합격 후에 생성한다.
4. **위임 현실 수용(Proxy-aware)**: CEO는 수행비서·실무 임원과 함께 학습하거나 일부를 위임한다. 1인 계정이 아니라 **CEO + 동반자(assistant)** 단위를 기본 모델로 설계한다.
5. **운영 자동화**: 지원 접수, 합격 통보, 과제 마감, 수료 안내 등 반복 커뮤니케이션은 알림톡·이메일로 100% 자동화한다.
6. **추천 기반 성장(Referral-first)**: CEO 시장의 유입은 검색이 아니라 지인 추천이다. 추천 루프를 1차 성장 엔진으로 설계한다.
7. **포탈 엔진의 재사용**: CAIO Portal, Gov Portal과 코어 패키지(packages/core, notify, vms)를 공유하는 모노레포 구조로 개발한다.
8. **기수별 대화 맥락 보존**: 카카오톡처럼 빠르게 묻고 답하되, 과제·공지·질문·알림 기록이 포탈 안에 남아 다음 운영과 AS에 재사용되도록 한다.

## 1.4 범위 (Scope)

- **포함**: 공개 사이트, 모집/지원, **추천 트래킹**, 선발 관리, **법인 결제(인보이스·입금확인·세금계산서 요청)**, 교육 CMS(**총 10회 주 1회 Zoom 정규 강의**, **Google Drive 기반 Zoom 녹화본 read-only 재생**, 과제, 출석, 공지), **기수별 대화방(과제·질문·알림 허브)**, **Google Drive 기반 대화방 파일 공유(모든 참여자 read/write)**, **AI/vibe coding/Claude Code/Claude Design 질문과 강의 영상 내용 질문에 답하는 AI 튜터**, **토요일 오프라인 보충 교육 세션(약 3회) 신청·안내·출석 관리**, **Build 결과물 쇼케이스**, **재학생 1:1 코칭 예약**, 수료생 AS 커뮤니티, **수료생 선택형 공개 프로필(사진·이력·회사소개, 완전 optional)**, **수료 90일 적용성과 추적**, AI 의사결정 브리프 콘텐츠, 카카오 알림톡/이메일 알림, 관리자 콘솔
- **제외 (v1.0 → v1.1 조정)**:
  - **온라인 결제(PG) 연동**은 제외하되, **법인 계좌이체 기반 인보이스·세금계산서 발행 요청·입금확인 플로우는 포함**(P0)으로 변경. 고가 CEO 프로그램의 현실 반영
  - 네이티브 모바일 앱(2단계, Tauri로 동일 코드 패키징 가능 구조만 유지)
  - 포탈 자체 실시간 화상 강의 기능(Zoom 링크·녹화·출석·알림 관리는 포함)
  - 영상 DRM(화면녹화 방지) — 비용 대비 v1 제외

## 1.5 과정 운영 모델 (v1.2)

AI4CEO 정규 과정은 **총 10회, 주 1회, 오후 6시~9시 Zoom 강의**를 기본 운영 모델로 한다. 포탈은 Zoom 자체를 대체하지 않고, 각 회차의 일정, Zoom 링크, 강의자료, 녹화 영상, 출석, 과제, 알림을 통합 관리한다.

단, AI4CEO의 바이브코딩 콘텐츠는 기술 변화와 수강생 수준에 맞춰 **기수마다 조금씩 바뀐다**. 따라서 포탈은 하나의 고정 커리큘럼을 모든 기수에 공유하지 않고, 기수 생성 시점에 **커리큘럼 템플릿을 복제해 해당 기수 전용 스냅샷**을 만든다. 이후 각 기수는 세션 제목, 일정, Zoom 링크, 실습 내용, 과제, 자료, 보충 세션 구성을 독립적으로 수정할 수 있어야 한다.

컴퓨터 사용이 익숙하지 않거나 정규 Zoom 실습을 따라오기 어려운 수강생을 위해, 과정 중 **토요일 오후 3시~6시 오프라인 보충 교육 세션을 약 3회** 제공한다. 이 보충 세션은 정규 강의의 대체재가 아니라 실습 격차를 줄이는 보완 장치이며, 포탈은 신청, 정원, 장소 안내, 참석 여부, 후속 과제 연결을 관리한다.

| 구분 | 운영 방식 | 포탈에서 관리할 것 |
|---|---|---|
| 정규 강의 | 총 10회, 주 1회, 18:00~21:00, Zoom | 일정, Zoom 링크, 자료, 녹화, 출석, 과제, 알림 |
| 오프라인 보충 세션 | 약 3회, 토요일 15:00~18:00, 대면 | 신청, 정원, 장소, 참석, 준비물, 후속 안내 |
| 1:1 코칭 | 필요 시 예약, 화면공유/대면 | 슬롯, 예약, 확정 알림, 코칭 메모 |
| 기수별 커리큘럼 | 템플릿 복제 후 기수별 수정 | 콘텐츠 버전, 세션/과제/자료 스냅샷, 변경 이력 |

# 2. 사용자 정의 (Personas & Roles)

## 2.1 페르소나

| 페르소나 | 설명 | 핵심 니즈 |
|---|---|---|
| P1. 지원자 (Prospect) | AI 도입을 고민하는 기업 CEO·임원. **지인 추천**으로 유입 | 과정이 나에게 맞는지 5분 안에 판단, 간편한 지원 |
| P2. 수강생 (Student) | 선발되어 교육 중인 CEO·임원. 시간이 절대적으로 부족 | 모바일에서 강의·과제를 한 화면에서 처리, **막힌 지점에서 즉시 도움**, **내 결과물 완성** |
| P2-a. 동반자 (Assistant) | 수강 CEO를 보조하는 수행비서·실무 임원 | CEO 대신/함께 자료·과제 처리, CEO에게 핵심만 전달 |
| P3. 수료생 (Alumni) | 과정을 마친 동문. 지속적 관계와 최신 정보 기대 | AI 의사결정 브리프, 질문 채널(AS), **폐쇄형 신뢰 네트워크** |
| P4. 운영자 (Admin) | 장동인 교수 및 운영 스태프 | 지원자 선발, 기수 운영, 콘텐츠 발행, **입금·세금계산서 처리**를 최소 클릭으로 |

## 2.2 역할 및 권한 (RBAC)

| 역할 | 권한 |
|---|---|
| `guest` | 공개 페이지 열람, 지원서 제출 |
| `applicant` | 본인 지원서 조회·수정(마감 전), 전형 상태 확인 |
| `student` | 소속 기수의 강의·자료·과제·공지 접근, 과제·결과물 제출, **기수별 대화방 참여**, Q&A 작성, **1:1 코칭 예약** |
| `assistant` | **연결된 CEO(student) 계정의 기수 콘텐츠 접근·과제 공동 작업, 기수별 대화방 참여.** CEO가 명시적으로 초대·연결. 결과물/제출은 CEO 명의로 귀속 |
| `alumni` | AS 커뮤니티, AI 의사결정 브리프 아카이브, **선택형 공개 프로필 작성/수정/비공개 전환**, (동의 시)동문 디렉토리, 오피스아워 예약 |
| `admin` | 전체 관리 (기수, 회원, 콘텐츠, 결제/입금, 알림, 통계) |

권한은 Supabase Auth + `profiles.role` + Row Level Security(RLS)로 강제한다. `student`는 `enrollments` 테이블 기준 본인 소속 기수(cohort) 데이터만 접근 가능하다. `assistant`는 `assistant_links(ceo_user_id)`로 연결된 CEO의 권한을 위임 상속하며, **권한 부여·해제는 CEO 본인만** 가능하다.

# 3. 사용자 여정 (End-to-End Journey)

```
[발견]      [탐색]        [지원]        [선발/결제]      [수강·Build]          [수료]        [동문]
지인추천 → 과정안내/FAQ → 지원서 제출 → 합격→인보이스   → 강의·과제·결과물완성  → 수료증/설문 → AS·브리프
 │            │              │(알림톡:접수) │(알림톡:합격)    │(알림톡:개강·마감)     │(알림톡:수료) │(90일 성과추적)
 │            └─ AI 의사결정 브리프 공개글이 유입 콘텐츠 역할                                    │
 └─ 추천 코드로 유입 추적 ──────────────────────────────────────── 동문 추천이 신규 기수 모집으로 순환 ┘
```

각 단계 전환은 시스템 이벤트로 기록되며, 이벤트마다 사전 검수된 알림톡 템플릿이 자동 발송된다. 알림톡 템플릿은 카카오 검수에 수 일이 소요되고 등록 후 수정이 불가하므로, 본 PRD 9장의 템플릿 초안을 개발 착수와 동시에 검수 신청한다.

**핵심 순환(Growth Loop)**: 수강생의 결과물(Build) → 쇼케이스 공유 → 동문 추천 코드 → 신규 지원자 유입 → 다음 기수. 이 루프가 SEO·뉴스레터보다 우선하는 1차 성장 엔진이다.

## 6.4 핵심 시퀀스 — 강의 영상 재생 (Google Drive read-only)

```
교수자/운영자 → (1) Zoom 녹화본 다운로드
             → (2) Google Drive 기수/세션 폴더에 업로드
             → (3) /admin 또는 /portal/sessions/[id]에서 Drive 파일을 세션 영상으로 연결
             → (4) Drive 권한: 수강생/assistant read-only, instructor/admin manage
             → (5) videos 레코드 생성 및 영상 공개
             → (6) T-14 알림톡/이메일/대화방 공지 발송
수강생/assistant → (7) /portal/sessions/[id]에서 영상 재생
```


## 6.6 핵심 시퀀스 — 대화방 파일 공유 (Google Drive)

```
참여자 → (1) /portal/chat 에 파일 업로드
       → (2) API Route: chat_room membership + write 권한 확인
       → (3) Google Drive: 기수별 Drive 폴더에 파일 업로드
       → (4) Drive 파일 권한을 대화방 참여자 read/write로 동기화
       → (5) chat_files + chat_messages(message_type=file) 저장
       → (6) 대화방에 파일 메시지 표시, 필요 시 알림톡/이메일 요약
```

## 6.7 핵심 시퀀스 — AI 질문 답변 튜터

```
수강생/assistant → (1) 대화방, AI 질문 패널, 또는 강의 영상 화면에서 질문 입력
                 → (2) API Route: cohort 권한 확인
                 → (3) 관련 커리큘럼, 과제, Q&A, Drive 파일 메타데이터, 강의 영상 요약/자막, 최근 대화 맥락 수집
                 → (4) OpenAI Responses API 호출 (기본: GPT-5.5 또는 운영 시점 승인 모델)
                 → (5) 답변 생성: 근거/불확실성/다음 행동 포함
                 → (6) ai_question_logs 저장, 사용자가 원하면 대화방에 ai_answer 메시지로 게시
                 → (7) 신고/재답변/강사 검수 요청 가능
```

# 12. Vibe UX 보강 아이디어

> 목적: 기존 PRD의 기능 범위를 크게 늘리지 않고, CEO가 "지원 → 첫 결과물 → 회사 적용 → 추천"까지 더 자연스럽게 이동하도록 UX 레이어를 보강한다. 아래 항목은 v1.1의 핵심 방향인 **결과물·회사 적용 중심**을 더 강하게 만드는 제안이다.

## 12.1 핵심 경험 재정의: "첫 결과물까지의 7일"

현재 PRD는 모집부터 수료 후 AS까지 End-to-End 구조가 잘 잡혀 있다. 다만 CEO 입장에서 가장 중요한 첫 aha moment는 "강의를 들었다"가 아니라 **내 회사에 쓸 수 있는 첫 결과물이 생겼다**는 감각이다. 따라서 포탈의 메인 경험을 다음 한 문장으로 더 좁힌다.

**지원부터 수강 첫 주까지, CEO가 자기 회사 문제 하나를 고르고 7일 안에 첫 Build 초안을 만든다.**

이를 위해 `/portal` 대시보드는 단순 강의 목록보다 **내 첫 Build 진행판**을 중심에 둔다.

- 오늘 해야 할 강의/과제보다 "내 결과물이 지금 어느 단계인가"를 먼저 보여준다.
- Build 단계는 `문제 선택 → 레퍼런스 보기 → 초안 만들기 → 코칭 받기 → 회사 적용 기록`의 5단계로 표시한다.
- 강의, 과제, 자료, 코칭은 각각 별도 메뉴가 아니라 Build 단계에 붙는 보조 행동으로 연결한다.
- 수료 기준의 `build_count`는 단순 등록 수가 아니라 "회사 적용 여부"와 함께 보여준다.

## 12.2 CEO용 첫 화면: Executive Command Center

CEO·임원 대상 서비스는 일반 LMS처럼 메뉴가 많은 구조보다, 매번 들어왔을 때 바로 판단할 수 있는 **요약형 작전판**이 더 적합하다.

P0 화면으로 `/portal` 첫 화면에 다음 4개 블록을 둔다.

| 블록 | 목적 | 대표 요소 |
|---|---|---|
| My Build | 이번 과정에서 만들 결과물의 진행 상태 | 단계, 다음 행동, 마지막 코칭 메모 |
| This Week | 이번 주 꼭 해야 할 것만 압축 | 강의 1개, 과제 1개, 마감, 장소/Zoom |
| Ask for Help | 막혔을 때 빠른 도움 요청 | 1:1 코칭 예약, Q&A, 운영팀 문의 |
| Delegate | 동반자에게 맡길 수 있는 일 | 자료 정리, 제출 초안, 일정 확인 |

핵심은 CEO에게 "무엇을 클릭할 수 있는가"보다 **지금 무엇을 결정하거나 위임하면 되는가**를 보여주는 것이다.

## 12.3 동반자(Assistant) UX 강화: 위임 가능한 일과 불가능한 일의 분리

`assistant` 역할은 PRD의 좋은 차별점이다. 여기에 UX 규칙을 더 명확히 둔다.

- Assistant는 CEO의 화면을 그대로 복제하지 않는다.
- Assistant 첫 화면은 `CEO에게 전달할 요약`, `대신 처리할 작업`, `CEO 확인이 필요한 결정`으로 나눈다.
- 결과물 제출은 CEO 명의로 귀속하되, 작성/수정 이력에는 Assistant 기여를 남긴다.
- CEO만 가능한 행동은 명확히 잠근다: 권한 해제, 공개 동의, 최종 제출 승인, 세금계산서 주요 정보 확정.
- Assistant가 작업을 마치면 CEO에게 "확인할 것 3개" 형태의 알림톡/이메일 요약을 보낸다.

이렇게 하면 "대신 해주는 사람"을 허용하면서도, 프로그램의 핵심 약속인 **CEO가 직접 자기 회사 적용점을 이해한다**는 기준이 흐려지지 않는다.

## 12.4 Build Brief: 결과물 등록 전 한 장짜리 문제 정의

Build 결과물 쇼케이스는 강력하지만, 처음부터 결과물을 올리라고 하면 부담이 크다. 결과물 전에 **Build Brief**를 둔다.

Build Brief는 수강생이 첫 주에 작성하는 1페이지 문제 정의다.

| 항목 | 질문 |
|---|---|
| 회사 문제 | 우리 회사에서 반복적으로 낭비되는 일은 무엇인가? |
| 사용자 | 이 결과물을 실제로 쓰거나 영향을 받는 사람은 누구인가? |
| 현재 방식 | 지금은 어떻게 처리하고 있는가? |
| 원하는 변화 | 이 결과물이 생기면 무엇이 빨라지거나 명확해지는가? |
| 성공 기준 | 2주 안에 무엇이 관찰되면 성공인가? |

Build Brief는 이후 과제, 코칭, 결과물 등록, 90일 성과 설문의 기준점이 된다. 즉 "무엇을 만들었는가"만이 아니라 **왜 만들었고 실제로 무엇이 바뀌었는가**를 추적할 수 있다.

## 12.5 결과물 쇼케이스의 두 가지 모드: 내부 학습용과 공개 성장용

PRD에는 기수 내 쇼케이스와 공개 쇼케이스가 이미 있다. 여기에 공개 부담을 낮추는 단계형 공개 UX를 추가한다.

- `cohort` 모드: 같은 기수 안에서만 보인다. 기본값.
- `alumni` 모드: 수료생 커뮤니티까지 보인다. 선택값.
- `public teaser` 모드: 제목, 문제 영역, 익명화된 성과만 공개한다. 회사명/화면/코드는 숨길 수 있다.
- `public full` 모드: 사용자가 명시 동의한 경우에만 상세 공개한다.

CEO 시장에서는 프라이버시가 기능이 아니라 신뢰의 전제다. 따라서 공개 쇼케이스는 "자랑"보다 **안심하고 공유 가능한 성과 카드**로 설계하는 편이 좋다.

## 12.6 90일 성과 추적을 설문이 아니라 "적용 리마인더"로 만들기

수료 후 90일 설문은 응답률이 낮아질 수 있다. UX상으로는 설문보다 **적용 리마인더**처럼 느껴져야 한다.

알림 문구의 방향:

- 나쁜 예: "90일 성과 설문에 응답해 주세요."
- 좋은 예: "그때 만든 결과물, 지금 회사에서 어디까지 쓰이고 있나요?"

90일 화면은 긴 설문이 아니라 3분 체크인으로 구성한다.

- 아직 안 씀: 이유 선택 + 다시 적용해볼 도움 요청
- 일부 사용 중: 누가 쓰는지, 어떤 점이 바뀌었는지 체크
- 잘 사용 중: 성과 수치/사례 입력 + 공개 범위 선택
- 추가 도움 필요: AS Q&A 또는 코칭 예약으로 연결

이 흐름은 KPI 측정뿐 아니라 alumni 재활성화와 심화 과정 추천으로 이어진다.

## 12.7 AI 의사결정 브리프를 "읽을거리"가 아니라 "결정 템플릿"으로 전환

AI 의사결정 브리프는 단순 콘텐츠보다 CEO의 행동을 유도하는 형식이 더 좋다. 각 브리프는 다음 구조를 기본 템플릿으로 한다.

1. 지금 왜 중요한가
2. 우리 회사에 해당되는 신호
3. CEO가 이번 주에 결정할 질문 3개
4. 바로 실험할 수 있는 작은 Build 아이디어
5. 관련 강의/결과물/동문 사례

이렇게 하면 콘텐츠가 포탈 체류용 글이 아니라, 다음 Build와 추천을 만드는 **의사결정 트리거**가 된다.
